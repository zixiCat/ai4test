# 为什么我选择这些模型

这份说明文档解释了：面向美国市场的 AI 产品，我建议采用哪些模型和服务区域，以及为什么这样选。

目标业务场景包括：

- 翻译工具
- 聊天机器人
- 命令调用与 Agent 式工作流
- 后续语音输入与语音输出能力

我们的应用服务对象虽然是美国市场，但模型端点并不一定必须部署在美国。只要其他区域能够提供更好的时延和更完整的模型覆盖，跨区域调用依然可能是更优解。为了让基准测试更接近真实生产链路，我使用 Promptfoo 汇总了 `output/bg`、`output/uk`、`output/use` 和 `output/usw` 中的多组缓存结果。

## 核心结论

我的推荐方案是：

- 模型服务区域：阿里云 DashScope 新加坡
- 主力交互模型：Qwen3.5-Omni-Plus
- 低时延回退模型：Qwen3.5-Omni-Flash
- 语音输出模型：Qwen3-TTS
- 语音输入模型：Qwen3-ASR

这是当前最适合我们项目的组合，因为它同时平衡了以下五个方面：

- 在所有缓存测试中都具备较好的交互式 TTFT
- TTFT 明显优于本次测试中的 DashScope 美国端点
- 相比 Qwen3.5-Flash 文本模型，输出长度更可控
- 对聊天、命令调用和后续多模态能力有更清晰的演进路线
- 文本、语音识别、语音合成都留在同一供应商体系内，集成复杂度更低

最重要的一点是：我并不推荐 DashScope 美国端点作为主路径。即便产品面向美国用户，我仍然推荐使用 DashScope 新加坡端点，因为实测 TTFT 更稳定、更快，而且新加坡区域的模型覆盖更完整。

## 为什么选择这套栈

价格当然重要，但不能只看文本 token 单价。

对这个项目而言，总成本至少包括：

- 模型调用成本
- 语音识别成本
- 语音合成成本
- 多供应商集成带来的工程成本
- 多 API 测试与可观测性成本
- 多平台并行带来的运维复杂度

因此，从实际项目成本看，统一的 Qwen 技术栈往往可能比“文本、ASR、TTS 分别选不同供应商”的混合方案更便宜，即使某个单独文本模型纸面价格看起来更低。

我们的产品路线图也不是纯文本应用。我们需要支持：

- 文本生成
- 翻译
- 对话式交互
- 工具调用与命令执行
- 语音转文本
- 文本转语音
- 后续多模态交互

这正是我在首个生产版本里优先考虑 Qwen 系列的原因。

推荐的能力分工如下：

| 能力                   | 推荐模型           |
| ---------------------- | ------------------ |
| 主聊天、翻译、命令调用 | Qwen3.5-Omni-Plus  |
| 低时延回退层           | Qwen3.5-Omni-Flash |
| 语音识别               | Qwen3-ASR          |
| 语音合成               | Qwen3-TTS          |

这样我们得到的是一套统一栈，而不是“文本一家、ASR 一家、TTS 再一家”的碎片化方案。

## 基准测试范围

我在这个工作区里测试了三类基准：

- 文本基准：`promptfooconfig.yaml`，使用 `prompts/ttft.txt` 和 `prompts/token-speed.txt`，覆盖新加坡 DashScope、美国 DashScope、OpenRouter 以及本地 XGD 提供方
- ASR 基准：`promptfoo.asr.yaml`，使用 `prompts/asr.txt` 和 `tests/asr-benchmark-cases.yaml`，当前通过 `qwen3-asr-flash singapore` 转写官方 DashScope `welcome.mp3` 示例音频
- TTS 基准：`promptfoo.tts.yaml`，使用 `prompts/tts.txt` 和 `tests/tts-benchmark-cases.yaml`，当前通过 `qwen3-tts-flash singapore` 的 `Cherry` 音色合成一条短英文样例和一条较长中文样例

这份中文文档同样纳入了 `output` 目录下全部 12 份缓存汇总文件：

- `output/bg/text.txt`、`output/bg/asr.txt`、`output/bg/tts.txt`
- `output/uk/text.txt`、`output/uk/asr.txt`、`output/uk/tts.txt`
- `output/use/text.txt`、`output/use/asr.txt`、`output/use/tts.txt`
- `output/usw/text.txt`、`output/usw/asr.txt`、`output/usw/tts.txt`

由于仓库里没有解释这些目录缩写分别对应什么环境或地域，我在文档里保留了原始目录 ID，不额外做未经验证的解释。

本轮基准主要统计以下指标：

- 平均总时延
- 平均 TTFT，也就是首 token 时间
- 平均每秒输出 token 数
- 平均 completion token 数
- 对 TTS 额外统计平均音频字节吞吐、平均音频字节数和平均输入字符数

如果目标是交互式产品，TTFT 是最关键的指标，因为它直接决定用户什么时候开始“感到系统在响应”。总时延同样重要，但会受到输出长度影响。至于 tokens/sec，在短回复场景里很容易出现异常偏高，所以这里只作为辅助参考。

## 如何复现

如果你想得到与这份文档可比较的数字，建议从美国机房或美国云主机执行，而不是直接从中国本地网络执行。这样网络路径会更接近真实美国生产环境。

1. 准备一台美国 Linux 机器，例如位于美国区域的 Vast.ai 实例。
2. 在 `.env` 中配置所需的 API Key。当前工作区可能涉及 `DASHSCOPE_SG_API_KEY`、`DASHSCOPE_US_API_KEY`、`OPENROUTER_API_KEY` 和 `SEETACLOUD_XGD_API_KEY`。
3. 依次运行 `npm run start:text`、`npm run start:asr` 和 `npm run start:tts`。
4. 查看生成的 `.promptfoo-output.json`、`.promptfoo-output.asr.json`、`.promptfoo-output.tts.json`，或直接读取 `output/<runner-id>/*.txt` 中的聚合汇总。

## 当前局限

这一轮测试主要衡量的是文本、ASR、TTS 三类能力的响应速度与流式表现，还没有完整覆盖以下质量维度：

- 真实业务翻译质量
- 工具调用或命令调用准确率
- 领域音频、口音、噪声环境下的转写准确率
- TTS 的自然度、发音质量与风格控制

下一轮建议补充更贴近业务的数据，包括翻译样例、客服多轮对话、工具调用任务、长音频与带噪音频，以及人工或自动化的语音质量评估。

### 关键观察

1. 在四组缓存结果里，Qwen3.5-Omni-Flash 一直是托管云侧 TTFT 最快的模型，因此非常适合做低时延回退层。
2. Qwen3.5-Omni-Plus 的 TTFT 虽然略高，但仍处在可接受范围内，而且输出长度显著更可控，因此更适合做默认生产模型。
3. Qwen3.5-Flash 新加坡版的问题不在于 TTFT，而在于 completion token 数在每次测试里都异常大，最终把总时延拉高到了不适合作为默认交互层的程度。
4. 四组缓存结果中，DashScope 美国版 flash 的 TTFT 都慢于新加坡版 flash，这说明区域选择必须依赖实测，而不是凭直觉假设。
5. Gemini 依然是一个有价值的外部文本基线，但当前决策不是“谁文本快一点”，而是“谁能提供一套更完整、更容易落地的生产栈”。
6. 本地 XGD 部署很快，但它更适合做工程参考基线，不适合直接作为这次托管生产路线的结论依据。

## 为什么我没有把 GLM、DeepSeek、Kimi、GPT 或 Gemini 平台作为主栈

这并不是因为这些模型不强。它们里有不少都非常有竞争力。

我这次没有把它们作为项目首发主栈，原因是这次决策标准是工程落地，而不是纯学术比较：

- 我需要一个模型家族同时覆盖文本、多模态、语音输入和语音输出
- 我希望基准结果尽量贴近目标部署路径
- 我希望生产栈尽可能复用已经在 Promptfoo 中验证过的 API 模式和评估路径
- 我希望降低工程、计费、监控和安全审查上的供应商碎片化成本
- 我希望第一版就能支撑聊天、翻译和命令调用，而不是先做一个更复杂的多供应商拼接架构

Gemini 依然是一个有价值的对照基线，我也建议继续保留至少一个顶级外部模型参与比较。但当前问题不是“谁单次文本生成快一些”，而是“谁能以更低复杂度提供一套完整、可上线、可扩展的文本 + ASR + TTS + 多模态能力栈”。在这个维度上，Qwen 更适合当前项目的首发版本。

至于 GLM、DeepSeek 和 Kimi，在这一阶段我没有看到足够大的业务收益，去证明我们应该在第一版就扩展供应商矩阵。更合理的做法是：先把第一版稳定在 Qwen 体系内，只有在第二家供应商能带来明确商业收益时再扩展。

## 为什么我不选择 DashScope 美国端点

我同样不建议当前把 DashScope 美国端点作为主服务区域。

原因很直接：

- 在四组缓存文本结果里，新加坡 DashScope flash 的 TTFT 都优于美国 DashScope flash
- DashScope 新加坡端点支持 Qwen Omni 系列、ASR 和 TTS
- DashScope 美国端点当前并不具备同等模型覆盖，因此无法承担我们希望的一体化能力栈

对这个项目来说，模型覆盖的完整性比“所有组件都在同一地理区域”更重要。应用本身仍然可以部署在美国，而模型调用走新加坡端点。至少在现阶段，这是更实际、更可靠的主路径。

## 最终建议

我的最终推荐是：

- 使用阿里云 DashScope 新加坡作为模型服务端点
- 使用 Qwen3.5-Omni-Plus 作为主产品模型
- 使用 Qwen3.5-Omni-Flash 作为低时延备选层
- 使用 Qwen3-TTS 作为语音合成模型
- 使用 Qwen3-ASR 作为语音识别模型

在 DashScope 美国端点没有拿出可比的 TTFT、并补齐 Omni、ASR、TTS 覆盖之前，不建议把它作为主路径。

在我们对真实业务提示词下的推理成本与时延表现更有把握之前，也不建议把 Qwen3.5-Flash 作为默认交互层。

一句话总结：当前最优方案是，面向美国用户提供服务，但调用阿里云新加坡端点，以 Qwen Omni + Qwen TTS + Qwen ASR 组成一套统一的多模态生产栈。
