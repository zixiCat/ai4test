# 为什么选择这些模型

这篇文档记录了我对我们面向全球市场的 AI 产品在模型选型和服务区域上的推荐，以及背后的决策逻辑。

我们需要支撑的业务场景包括：

- 翻译工具
- 聊天机器人
- 命令调用与 Agent 工作流
- 语音输入和语音输出

产品面向全球用户，模型服务节点的选择应以延迟和模型覆盖为准，而不是以应用部署的地理位置为准。为了让基准测试覆盖不同地区的访问路径，我用 Promptfoo 从四个地点采集了测试数据，分别存放在 `output/bg`、`output/uk`、`output/use` 和 `output/usw` 目录下。

## 推荐结论

综合所有测试结果，我的推荐如下：

- 模型服务区域：阿里云 DashScope 新加坡节点
- 主交互模型：Qwen3.5-Omni-Plus
- 低延迟备选模型：Qwen3.5-Omni-Flash
- 语音合成：Qwen3-TTS
- 语音识别：Qwen3-ASR

这个方案能同时满足以下五点要求：

- 在全部测试节点上都有稳定的交互 TTFT 表现
- TTFT 优于 DashScope 美国节点
- 输出可控性比 Qwen3.5-Flash 文本模型更好
- 为后续聊天机器人和命令工作流提供清晰的多模态升级路径
- 文本、语音合成、语音识别统一在同一供应商，集成成本低

有一点需要特别说明：**我推荐的是新加坡 DashScope 节点，而不是美国节点**——哪怕产品是面向美国用户的。原因是实测下来新加坡节点的 TTFT 始终更低，而且模型覆盖更完整。

## 为什么选这套技术栈

成本很重要，但不能只看文本 Token 的单价。

对这个项目来说，真实的总成本还包括：

- 模型调用费用
- 语音识别费用
- 语音合成费用
- 多供应商集成带来的工程开发成本
- 跨多套 API 的测试和可观测性成本
- 不同能力分散在不同平台时的日常运维负担

因此，用统一的 Qwen 技术栈，实际下来可能比拼凑多个供应商的方案更省钱，哪怕某个单独的文本模型在报价上看起来更便宜。

我们的产品路线图也不是单纯的文本场景，需要覆盖的能力包括：

- 文本生成
- 翻译
- 对话交互
- 工具调用与命令执行
- 语音转文字
- 文字转语音
- 以及未来的多模态交互

这是我在第一次正式上线时选择 Qwen 全家桶的核心原因。

各能力的推荐模型如下：

| 能力                         | 推荐模型           |
| ---------------------------- | ------------------ |
| 主聊天机器人、翻译、命令调用 | Qwen3.5-Omni-Plus  |
| 低延迟备选层                 | Qwen3.5-Omni-Flash |
| 语音识别                     | Qwen3-ASR          |
| 语音合成                     | Qwen3-TTS          |

这样整套技术栈是统一的，而不是文本、ASR、TTS 各用一家供应商。

## 基准测试范围

本次一共测试了三组基准：

- **文本基准**：使用 `promptfooconfig.yaml`，搭配 `prompts/ttft.txt` 和 `prompts/token-speed.txt`，覆盖新加坡 DashScope、美国 DashScope、OpenRouter 和本地 XGD 供应商
- **ASR 基准**：使用 `promptfoo.asr.yaml`，搭配 `prompts/asr.txt` 和 `tests/asr-benchmark-cases.yaml`，通过 `qwen3-asr-flash singapore` 转录 DashScope 官方的 `welcome.mp3` 示例音频
- **TTS 基准**：使用 `promptfoo.tts.yaml`，搭配 `prompts/tts.txt` 和 `tests/tts-benchmark-cases.yaml`，用 Cherry 音色分别合成一段短英文和一段较长的中文

`output` 目录下共有十二份缓存摘要，来自四个测试节点：

- `bg` — 保加利亚
- `uk` — 英国
- `use` — 美国东部
- `usw` — 美国西部

每组测试采集了以下指标：

- 平均总延迟
- 平均 TTFT（首 Token 时间）
- 平均每秒 Token 数
- 平均完成 Token 数
- TTS 专项：平均每秒音频字节数、平均音频字节数、平均输入字符数

对于交互式产品来说，**TTFT 是最关键的指标**，它直接决定了用户感受到的响应速度。总延迟同样重要，但它会随输出长度变化，需要结合完成 Token 数一起看。每秒 Token 数是次要参考，在短输出场景下这个值容易虚高。

## 如何复现

如果你想得到可以和本文对比的数据，请在接近目标用户的服务器上跑测试，而不是从国内网络跑。本次基准选择了美国节点作为主要参考，因为美国是核心用户群之一，但建议同时从欧洲或东南亚节点补充测试，以覆盖全球访问路径。

1. 准备一台在目标地区的 Linux 机器，比如 Vast.ai 上的美国或欧洲实例。
2. 在 `.env` 里填好所需的 API 密钥，本项目可能用到 `DASHSCOPE_SG_API_KEY`、`DASHSCOPE_US_API_KEY`、`OPENROUTER_API_KEY` 和 `SEETACLOUD_XGD_API_KEY`，按需填写。
3. 依次运行 `npm run start:text`、`npm run start:asr` 和 `npm run start:tts`。
4. 查看生成的 `.promptfoo-output.json`、`.promptfoo-output.asr.json` 和 `.promptfoo-output.tts.json`，或者直接看 `output/<runner-id>/*.txt` 里的聚合摘要。

## 当前的不足

本轮测试主要考察的是平台响应速度和流式传输表现，以下方面还没有充分覆盖：

- 真实业务内容的翻译质量
- 工具调用和命令执行的准确率
- 领域音频、口音、噪声环境下的转录准确率
- TTS 的自然度、发音质量和风格可控性

下一轮测试应该补充：面向翻译、客服对话、工具调用的业务专项提示词，更长的领域音频样本，以及对语音输出自然度和发音质量的人工或自动评估。

### 几个关键发现

1. Qwen3.5-Omni-Flash 在全部四个节点上都拿到了托管云里最低的 TTFT，是延迟优先场景的最佳选择。
2. Qwen3.5-Omni-Plus 的 TTFT 与之相差不大，同时输出长度更可控，更适合作为默认的生产层。
3. 新加坡的 Qwen3.5-Flash 在 TTFT 上没有问题，但每次运行都会产生大量完成 Token，导致总延迟完全无法用于交互场景。
4. 美国 DashScope Flash 节点在每个测试节点上都慢于新加坡节点——选择服务区域要靠实测，不能想当然。
5. Gemini 作为外部文本基线仍然有参考价值，但我们这里的决策是选整套生产栈，不只是比文本速度。
6. 本地 XGD 部署速度很快，但我把它当作工程参考，而不是生产选项，因为运营模式完全不同。

## 竞品参考：实时语音 AI 平台

两个值得对比的实时语音 AI 平台：

豆包 Live（字节跳动）和 Gemini Live（谷歌）在交互语音场景下的首次响应时间大约在 **800 ms 到 1300 ms** 之间。这说明我们这次基准里 Qwen3.5-Omni-Flash 在四个节点的表现（767 ms 到 1390 ms）已经处于行业主流水平，在原始首响延迟上具有竞争力。

不选豆包 Live 或 Gemini Live 的原因和后面说的一样：供应商碎片化。这两个平台的核心产品是端到端的语音对话，而不是我们翻译工具、命令调用和自定义聊天机器人所需要的那套灵活的文本加语音 API。引入它们意味着语音和文本要各维护一套集成，工程成本和运维复杂度都会增加。

## 为什么不选 GLM、DeepSeek、Kimi、GPT 或 Gemini 作为主栈

不是因为这些模型不行，有些还相当优秀。

但我在这个项目里做技术选型，看的是实际问题，不是跑分排名：

- 我需要一个同时支持文本、多模态、语音输入和语音输出的模型家族
- 我需要测试结果已经贴近真实部署路径的方案
- 我需要生产栈和我们已经用 Promptfoo 跑通的 API 模式保持一致
- 我需要减少工程、计费、监控、安全审查各个环节的供应商数量
- 我需要一个不用先搭复杂的多供应商架构就能支撑聊天、翻译和命令工作流的方案

Gemini 作为外部基准对照仍然有用，我会继续在对比集里保留一个主流外部模型。但这里要回答的问题不只是"哪个文本模型最快"，而是"哪套栈能同时给我一个稳定的默认文本模型、一个快速备选层，还有 ASR 和 TTS，运维路径也足够清晰"。在这个维度上，Qwen 是第一版上线的更好选择。

对于 GLM、DeepSeek 和 Kimi，在第一版上线阶段，我看不到值得为它们扩展供应商矩阵的理由。我更倾向于先把第一版统一在 Qwen 技术栈上，等有具体的业务理由再考虑引入第二家供应商。

## 为什么不选美国 DashScope 节点

我同样不推荐美国 DashScope 节点作为主要服务区域。

原因很直接：

- 四组缓存文本测试里，新加坡 DashScope Flash 节点的 TTFT 全部优于美国节点
- 新加坡节点支持 Qwen Omni 系列、ASR 和 TTS
- 美国节点目前模型覆盖不完整，无法支撑我们需要的完整技术栈

对这个项目来说，模型覆盖的完整性比把所有组件都放在同一个地理位置更重要。应用跑在美国、调用新加坡的端点，在工程上完全可行，所以现阶段的实际选择就是用新加坡，等美国节点的能力和性能都跟上来再说。

## 最终结论

综合以上分析，我的推荐是：

- 模型服务端点：阿里云 DashScope 新加坡
- 主力产品模型：Qwen3.5-Omni-Plus
- 延迟敏感场景备选：Qwen3.5-Omni-Flash
- 语音合成：Qwen3-TTS
- 语音识别：Qwen3-ASR

在美国 DashScope 节点达到同等 TTFT 水平、并且完整支持 Omni、ASR 和 TTS 之前，不要把它作为主要服务路径。

在我们用真实业务提示词跑通、对推理成本和延迟行为有足够把握之前，不要把 Qwen3.5-Flash 作为默认交互层。

一句话总结：**面向全球用户的产品，调用阿里云新加坡节点，用 Qwen Omni + Qwen TTS + Qwen ASR 作为一套完整的集成技术栈——这是目前最合适的选择。**
